The Problem

The collections operations team at Vidyuttech was spending over 20 hours every week on a fully manual reporting workflow. Each Monday, analysts would pull raw data from 10+ PostgreSQL tables, paste it into Excel, reconcile column mismatches by hand, and format summary reports for leadership.

Beyond the time cost, the process was fragile. A single misaligned join or copy-paste error would go undetected until leadership spotted a discrepancy in a review meeting — often days later. There was no audit trail, no validation, and no way to trace a wrong number back to its source.

"We were spending more time assembling the data than actually reading it. Every week it felt like we were starting from scratch."

Requirements Gathering

Before writing a single line of code, I spent two sessions with the operations lead mapping every manual step in their existing process. Key questions I asked:

  • Which tables are joined, and what are the common keys?
  • What transformation rules are applied (aggregations, filters, business logic)?
  • What does "correct" look like — are there known totals or benchmarks to validate against?
  • Who consumes the report and in what format?
  • What's the acceptable latency — do they need real-time or is daily sufficient?

This scoping revealed that the core logic was stable — the same ~12 transformations ran every time — but nobody had ever written it down. Documenting the business logic before automating it was the most important step.

Technical Approach

I designed a three-stage Python ETL pipeline, scheduled via a daily cron job, that mirrored exactly what the analysts were doing manually — but with validation baked in at every stage.

1
Extract

Python + psycopg2 queries 10+ PostgreSQL tables with parameterised date windows, pulling only the delta since last run to keep query times low.

2
Transform & Validate

Pandas applies the 12 documented business-logic transformations. A validation layer then checks row counts, null thresholds, and expected totals against historical baselines. Any anomaly triggers an email alert before the report is generated.

3
Load & Deliver

Clean output is written back to a PostgreSQL summary table (powering Metabase dashboards) and simultaneously exported as a formatted Excel file, auto-emailed to stakeholders each morning at 7 AM.

Key Challenges

  • Undocumented edge cases — three business rules only surfaced after the first run exposed mismatch. I added them to the transform layer and documented them in a shared Notion page so future analysts could maintain the pipeline.
  • Schema drift — one upstream table added a new column mid-project. I built a lightweight schema-diff check into the extract stage so the pipeline fails loudly rather than silently ingesting wrong data.
  • Stakeholder trust — the team was sceptical of automation at first. I ran the pipeline in parallel with the manual process for two weeks, producing side-by-side comparisons, before switching over fully.

Results

After a two-week parallel validation period, the manual workflow was retired entirely. The pipeline has run daily without a missed report since deployment.

  • Reporting time cut from 20+ hours/week to under 2 hours (monitoring + exception handling)
  • 99% daily accuracy — anomaly alerts catch data issues before they reach leadership
  • Zero missed Monday reports in the 6 months post-deployment
  • The Metabase dashboard downstream received 40% fewer ad-hoc data requests

What I'd Do Differently

I'd push for a proper data contract with the upstream teams earlier. The schema drift incident could have been avoided with a shared agreement on column stability. I'd also add dbt-style lineage documentation from the start, rather than retroactively — it makes onboarding the next analyst significantly faster.
