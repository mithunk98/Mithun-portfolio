The Problem

The finance team was running monthly reconciliation jobs that processed several million rows of transaction data across multiple fiscal periods. The existing tooling — a combination of Excel macros and single-threaded Python scripts — took approximately 4 hours per full run, completely blocking the analyst's machine for the duration.

This bottleneck meant finance could only run full reconciliations once per reporting cycle. Any mid-month checks had to be done on subset samples, introducing uncertainty into interim reports.

"We couldn't afford to run this more than once a month. We needed something that could process the full dataset on demand, not overnight."

Profiling the Bottleneck

Before proposing Hadoop, I profiled the existing scripts to understand where the time was actually going:

  • ~60% of runtime was a large multi-table JOIN across 3 fiscal period tables — entirely single-threaded
  • ~25% was a rolling aggregation that recalculated from scratch each run rather than incrementally
  • ~15% was file I/O — reading and writing large CSVs sequentially

The JOIN and aggregation steps were both embarrassingly parallel — ideal Map-Reduce candidates. The I/O issue would be resolved naturally by moving to HDFS distributed storage.

Architecture

1
Data Ingestion to HDFS

Migrated source CSVs and PostgreSQL exports to HDFS. Partitioned files by fiscal period to enable selective processing — so a single-month run only reads that month's partition rather than the full dataset.

2
Map Phase — Parallel Key Extraction

Mapper functions extract transaction keys and fiscal period identifiers, distributing records across nodes by account group. Each mapper processes its shard independently with zero inter-node dependency.

3
Reduce Phase — Aggregation & Reconciliation

Reducer functions apply the reconciliation logic (sum, variance checks, period-over-period delta) per account group in parallel. Results written back to HDFS then exported to PostgreSQL summary tables.

4
Validation Layer

Post-run validation script checks grand totals against known control sums, flagging any reconciliation breaks before results are surfaced to the finance team.

Key Challenges

  • Data skew — a small number of large corporate accounts produced disproportionately large shards, causing some reducers to run 3× longer than others. Resolved by adding a custom partitioner that distributed by account size bucket rather than account ID.
  • Environment setup — spinning up a local Hadoop pseudo-distributed cluster for development took significant configuration. Documented the full setup process so the finance team could run it independently.
  • Legacy output format — the downstream reporting tool expected specific CSV column ordering. Added a post-processing step to reformat HDFS output before delivery.

Results

  • Processing time reduced from ~4 hours to ~45 minutes — 81% improvement
  • Finance team can now run full reconciliations on demand, not just monthly
  • Custom partitioner resolved data skew — all reducers now complete within 3 minutes of each other
  • Legacy single-threaded scripts fully decommissioned

What I'd Do Differently

For a production deployment I'd move to a managed service like AWS EMR or Databricks rather than a self-managed Hadoop cluster — the operational overhead of cluster management isn't justified at this data volume. I'd also replace the manual HDFS ingestion with a scheduled Airflow DAG to keep the pipeline fully automated end-to-end.
